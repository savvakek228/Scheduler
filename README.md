# Scheduler

Веб-приложение для **отложенного выполнения фоновых задач** с надёжной доставкой: паттерн **transactional outbox**, **inbox** для защиты от дублей, **повторы с экспоненциальной задержкой** и **dead letter** после исчерпания попыток. Есть **HTTP API** и **Blazor Server** UI.

## Назначение

Вы создаёте задачу с полезной нагрузкой (строка, обычно JSON), временем запуска `RunAt` и опциональным **ключом идемпотентности**. Система гарантирует, что задача будет обработана **хотя бы один раз** (семантика *at-least-once*); повторная доставка та же по смыслу дедуплицируется по **идентификатору сообщения** в inbox.

## Архитектура решения

| Проект | Роль |
|--------|------|
| **Scheduler.Domain** | Сущности: задача, outbox, inbox, dead letter |
| **Scheduler.Contracts** | DTO для API и UI |
| **Scheduler.Application** | Сценарии: планирование, обработка сообщений, outbox relay, запросы |
| **Scheduler.Infrastructure** | EF Core + SQLite, канал обработки, реализация `ITaskExecutor` |
| **Scheduler.App** | ASP.NET Core: Minimal API + Blazor Server |

При старте создаётся файл SQLite (см. ниже) и схема БД через `EnsureCreatedAsync` — отдельные миграции не используются.

## Как устроен поток выполнения

### 1. Планирование (`POST /api/tasks` или UI «Создать»)

- Проверяется **идемпотентность**: при переданном `IdempotencyKey`, если задача с таким ключом уже есть, возвращается существующий `taskId` без создания дубликата.
- В **одной транзакции** сохраняются:
  - строка **задачи** со статусом `Pending` и версией для оптимистичной блокировки;
  - сообщение **outbox** со статусом «ожидает публикации» и JSON-конвертом (`taskId`, `payload`).

Так достигается согласованность: либо есть и задача, и событие на её обработку, либо при сбое откатывается всё.

### 2. Outbox relay (фоновый сервис)

Циклически (с небольшим интервалом) читаются **неопубликованные** сообщения outbox. Для каждого:

1. Во внутренний **канал** кладётся конверт (`messageId`, `taskId`, тип, JSON).
2. Outbox помечается опубликованным.
3. Статус задачи переводится в `Queued` (если ещё был `Pending`).

Так обработка отвязана от HTTP-запроса создания задачи.

### 3. Обработчик задач (фоновый сервис)

Читает конверты из канала и для каждого вызывает **обработчик сообщений**:

1. **Inbox**: вставка по `messageId`. Если запись уже есть — это повторная доставка, обработка прекращается (идемпотентность на уровне сообщения).
2. Загружается задача; если её нет — inbox очищается и выход.
3. Если **`RunAt` ещё в будущем** — задача **откладывается** (`NextAttemptAt`, статус остаётся в очереди), в outbox добавляется новое сообщение на повторную попытку с тем же payload. Когда наступит время, повторный прогон сможет выполнить работу.
4. Через **оптимистичную блокировку** (совпадение `Version` и допустимый статус / окно `NextAttemptAt`) задача переводится в **`Running`**.
5. Вызывается **`ITaskExecutor`** (по умолчанию — `DefaultTaskExecutor`): парсит JSON, при необходимости симулирует ошибки по полю `failUntilAttempt` (см. ниже).
6. При успехе — статус **`Completed`**. При исключении — либо **повтор** с экспоненциальной задержкой и новым outbox, либо перенос в **dead letter**, если попыток не осталось.

### 4. Повторы и dead letter

- **Максимум попыток** и **базовая задержка** задаются в `RetryPolicyOptions` (по умолчанию до 5 попыток, базовый шаг 2 с; множитель — степень двойки от номера попытки).
- После исчерпания попыток задача помечается как dead letter, в таблицу dead letter пишется причина и число финальных попыток.

### 5. Повторная постановка из dead letter

`POST /api/dead-letters/{id}/replay` (и кнопка в UI) удаляет запись dead letter, сбрасывает счётчики попыток у задачи, ставит её в очередь и снова создаёт сообщение outbox — задача снова проходит обычный цикл.

## Состояния задачи

`Pending` → `Queued` → `Running` → `Completed`, либо ветка на **повтор** (снова `Queued` с `NextAttemptAt`), либо **`DeadLetter`**.

## Встроенный исполнитель (`DefaultTaskExecutor`)

Ожидает JSON конверта с полями `taskId` и `payload` (как при создании задачи через API).

- Для **тестов сбоев** в корне JSON или во вложенном JSON `payload` можно указать **`failUntilAttempt`** (целое число): попытки с номером **меньше** этого значения завершаются исключением.

## HTTP API (префикс `/api`)

| Метод | Путь | Назначение |
|--------|------|------------|
| POST | `/tasks` | Создать задачу (тело: payload, runAt, опционально idempotencyKey) |
| GET | `/tasks` | Список последних задач (`take`) |
| GET | `/tasks/{id}` | Детали задачи |
| GET | `/dead-letters` | Список dead letter |
| POST | `/dead-letters/{id}/replay` | Вернуть задачу в очередь |
| GET | `/diagnostics/outbox` | Диагностика сообщений outbox |

## Веб-интерфейс (Blazor Server)

- **Главная** — вводная страница.
- **Задачи** — список и переход к деталям.
- **Создать** — форма создания задачи.
- **Dead letter** — список и replay.
- **Outbox** — просмотр очереди публикации (диагностика).

## Запуск и конфигурация

Требуется **.NET 9 SDK** (или соответствующий runtime для опубликованного приложения).

```bash
cd src/Scheduler.App
dotnet run
```

По умолчанию база: **`scheduler.sqlite`** в каталоге контента приложения (`ContentRootPath`). Путь можно переопределить конфигурацией:

```json
{ "Scheduler": { "SqlitePath": "C:\\data\\my-scheduler.sqlite" } }
```

Либо переменная окружения в стиле ASP.NET: `Scheduler__SqlitePath`.

Интеграционные тесты подставляют свой временный файл через ту же настройку.

## Тесты

```bash
dotnet test Scheduler.sln
```
